# Branching strategie
We werken met gitflow. Feature branches worden aangemaakt gebaseerd op develop, en pas weer teruggehaald naar develop wanneer ze klaar en getest zijn. Daarna wordt develop voor de zekerheid meteen getest.
Aan het eind van de sprint testen we weer develop, en wordt een pull request naar main gemaakt. Deze uiteindelijke oplevering krijgt een versie tag. Als er iets fout is, kunnen hotfix/ branches worden aangemaakt om snel problemen op te lossen.

Wanneer de software eenmaal burikbaar is zal een release branch aangemaakt worden en zal dat worden gemaakt tot een github release. 

# Werkwijze documentatie & tijdschrijfformulier
Om een massaal aantal branches en pulls te vermijden, werk deze direct bij in develop.

Een leerapplicatie ontwikkelen waarmee gebruikers op een leuke, motiverende en foutloze manier leren typen, ondersteund door realtime feedback en gepersonaliseerde oefeningen.
